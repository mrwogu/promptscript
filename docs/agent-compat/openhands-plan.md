## Research Validation

### Output Path

The research states the formatter writes to `.openhands/rules/project.md`. Code inspection confirms:

```ts
// packages/formatters/src/formatters/openhands.ts
outputPath: '.openhands/rules/project.md',
dotDir: '.openhands',
```

This is correct. The `buildVersions` helper in `create-simple-formatter.ts` derives skills output as `.openhands/skills/<name>/SKILL.md` for `full` mode, which the research documents as valid (if deprecated per current OpenHands docs that prefer `.agents/skills/`).

### Formatter Structure

The formatter is a minimal one-liner delegation to `createSimpleMarkdownFormatter`. This is accurate: the file is 12 lines with no overrides. The factory wires up:

- `hasSkills = true` (default) â€” skill files generated in `full` mode via `generateSkillFile`
- `hasAgents = false` (default) â€” no agent sub-files
- `hasCommands = false` (default) â€” no command files
- `skillFileName = 'SKILL.md'` (default) â€” matches AgentSkills standard

### Feature Matrix Status

Confirmed: `openhands` appears only in the `ToolName` union type (line 47 of `feature-matrix.ts`). It has **no entries** in any `FeatureSpec.tools` record. Every feature row in `FEATURE_MATRIX` omits `openhands` entirely. The research claim that "all statuses are derived by cross-reference" is accurate â€” there is no formatter-enforced mapping in the codebase.

### Skill Triggers Gap

The research identifies that `generateSkillFile` in `markdown-instruction-formatter.ts` emits only `name` and `description` in YAML frontmatter (lines 380-383):

```ts
lines.push('---');
lines.push(`name: ${config.name}`);
lines.push(`description: ${this.yamlString(config.description)}`);
lines.push('---');
```

The `MarkdownSkillConfig` interface has no `triggers` field. The `extractSkills` method does not read a `triggers` property from the AST `skills` block. This confirms the gap: `triggers` arrays cannot currently be emitted in SKILL.md frontmatter for any formatter, not just OpenHands.

### Skills Path Gap

The research notes that OpenHands docs now recommend `.agents/skills/<name>/SKILL.md` while the formatter uses `.openhands/skills/`. This is confirmed by the `dotDir: '.openhands'` configuration. The research correctly characterises this as low priority because OpenHands still discovers `.openhands/skills/` (it is in the fallback discovery order).

### `AGENTS.md` Gap

The research flags that `AGENTS.md` at the repository root is the canonical OpenHands always-on file. The formatter does not generate this file. This is consistent: the formatter is responsible for the `.openhands/rules/project.md` path, and `AGENTS.md` is a developer-maintained file. The research correctly rates this low priority.

### Version Description Accuracy

The `multifile` version description produced by `buildVersions` for a nested path is:

```
Single .openhands/rules/project.md file (skills via full mode)
```

This is generated by the `isNested` branch in `buildVersions`. The research observation is accurate: the description implies skills appear in `multifile` mode, but the `formatMultifile` method only adds skills when `skillsInMultifile` is true, which defaults to `false`. The description is mildly misleading.

### Research Accuracy Summary

All research claims have been validated against the code. No factual errors found. Priority assessments are appropriate.

### P2 â€” Skill Triggers Support (medium priority, cross-formatter improvement)

**Files affected:**

- `packages/formatters/src/markdown-instruction-formatter.ts` â€” `MarkdownSkillConfig` interface and `extractSkills` / `generateSkillFile` methods
- Any formatter spec files that test skill output

The `MarkdownSkillConfig` interface must gain an optional `triggers` field:

```ts
export interface MarkdownSkillConfig {
  name: string;
  description: string;
  content: string;
  triggers?: string[];
  resources?: Array<{ relativePath: string; content: string }>;
}
```

`extractSkills` must read `triggers` from the AST skill object:

```ts
triggers:
  obj['triggers'] && Array.isArray(obj['triggers'])
    ? (obj['triggers'] as Value[]).map((v) => this.valueToString(v))
    : undefined,
```

`generateSkillFile` must emit the `triggers` array in YAML frontmatter when present:

```ts
lines.push('---');
lines.push(`name: ${config.name}`);
lines.push(`description: ${this.yamlString(config.description)}`);
if (config.triggers && config.triggers.length > 0) {
  lines.push('triggers:');
  for (const trigger of config.triggers) {
    lines.push(`  - ${this.yamlString(trigger)}`);
  }
}
lines.push('---');
```

This is a base-class change. It improves all formatters that use `generateSkillFile` (OpenHands, Goose, Windsurf, Cline, Roo, Augment, Kiro, and others). It requires a corresponding PromptScript language addition: the `skills` block must support a `triggers` property on each skill entry. This is a parser/AST concern outside the formatter package scope and must be coordinated with the parser team before this change is implemented.

### P3 â€” Skills Path Alignment (low priority, optional)

**File:** `packages/formatters/src/formatters/openhands.ts`

Change `dotDir` from `.openhands` to `.agents` to align with the current OpenHands recommended skill path:

```ts
// Before
dotDir: '.openhands',

// After
dotDir: '.agents',
```

This would move skill output from `.openhands/skills/<name>/SKILL.md` to `.agents/skills/<name>/SKILL.md`. The primary rules file path (`.openhands/rules/project.md`) is controlled by `outputPath` and would be unaffected.

**Risk:** This is a breaking change for any users already deploying skills via `.openhands/skills/`. It should only be made if OpenHands officially removes the `.openhands/skills/` discovery fallback or explicitly deprecates it in a way that causes agent failures.

**Recommendation:** Defer until OpenHands removes the legacy discovery path. Add a JSDoc note in `openhands.ts` documenting the discrepancy instead.

### P4 â€” `AGENTS.md` Generation Option (low priority, optional)

**File:** `packages/formatters/src/formatters/openhands.ts`

An `AGENTS.md` generation mode is theoretically possible by switching `outputPath` to `AGENTS.md` and `dotDir` to `.agents`. This would require `buildVersions` to produce a non-nested description path since `AGENTS.md` does not start with `.agents/`.

The `buildVersions` logic already handles the non-nested case (the `!isNested` branch), so this variant is mechanically feasible using `createSimpleMarkdownFormatter` with no code changes to the factory.

**Recommendation:** Treat this as a future formatter variant (`openhands-agents`) rather than a change to the existing `openhands` formatter, to avoid breaking the existing output path.

### P5 â€” Multifile Version Description Fix (cosmetic, low priority)

**File:** `packages/formatters/src/create-simple-formatter.ts`

The `buildVersions` function generates this description for the `multifile` version when `outputPath` is nested:

```ts
const multifileDesc = isNested
  ? `Single ${outputPath} file (skills via full mode)` // misleading
  : `${outputPath} + ${dotDir}/skills/<name>/SKILL.md`;
```

The parenthetical `(skills via full mode)` is present for all nested-path formatters, not just OpenHands. Changing it affects every formatter using this factory. A clearer description:

```ts
const multifileDesc = isNested
  ? `Single ${outputPath} file`
  : `${outputPath} + ${dotDir}/skills/<name>/SKILL.md`;
```

This is a cosmetic fix with no runtime impact. It can be done independently.

## Implementation Notes

### Ordering

1. Implement P1 (feature matrix) first â€” it is self-contained and unblocks accurate coverage reporting.
2. Implement P5 (version description fix) alongside P1 since both are in the formatter package and low risk.
3. Coordinate P2 (skill triggers) with the parser team as a separate issue. It requires: (a) parser support for `triggers` property in `skills` blocks, (b) AST type updates in `@promptscript/core`, (c) base class changes in `markdown-instruction-formatter.ts`, (d) test coverage for the new YAML frontmatter output.
4. Defer P3 and P4 until OpenHands platform evolution clarifies the path.

### Testing for P1

Feature matrix entries are exercised by the existing matrix coverage tests. Adding `openhands` to each `FeatureSpec.tools` record will make coverage tests include OpenHands in their reporting. No new test files are needed for P1.

### Testing for P2

When P2 is implemented, the following tests are required:

- Unit test in `markdown-instruction-formatter.spec.ts`: skill with `triggers` array emits correct YAML frontmatter.
- Unit test: skill without `triggers` emits clean two-field frontmatter (regression guard).
- Integration test in `openhands.spec.ts`: end-to-end SKILL.md output with triggers from a `.prs` fixture.

### No Formatter Code Changes for Current Scope

The `openhands.ts` formatter file itself requires no changes for P1, P2, or P5. It correctly delegates to the factory and the base class. Only P3 (dotDir change) touches `openhands.ts` directly.

### Comparison to Similar Formatters

The OpenHands formatter is structurally identical to `goose.ts` (same factory, same default options, different `outputPath` and `dotDir`). Any improvements to the base class or factory will automatically benefit both. When implementing P2, Goose should be tested alongside OpenHands since it shares the same SKILL.md generation path.
