## Summary

iFlow CLI is a terminal-embedded AI coding assistant developed by an Alibaba-affiliated team. It supports multiple AI models (Qwen3-Coder, Kimi K2, DeepSeek v3, and OpenAI-compatible endpoints) and positions itself as an open-source alternative to tools like Claude Code. The CLI uses an `IFLOW.md` file at the project root as its primary codebase documentation artifact (generated by `/init`), but also maintains a `~/.iflow/` user-level configuration directory. No formal spec for project-level rule files was found in the official documentation; the platform does not appear to advertise a dedicated project rules location equivalent to `.cursor/rules` or `.windsurf/rules`. The PromptScript formatter targets `.iflow/rules/project.md`, which is a reasonable convention consistent with the tool's dot-directory pattern, but this path is not officially documented as a rules-loading location.

## Instruction File Format

### Primary project documentation file

| Property     | Value                                                   |
| ------------ | ------------------------------------------------------- |
| Filename     | `IFLOW.md`                                              |
| Format       | Plain Markdown                                          |
| Location     | Project root (generated by `/init` command)             |
| Schema       | None — free-form Markdown, generated from codebase scan |
| Front matter | Not documented                                          |

The `/init` command scans the repository structure, learns architecture and dependencies, and writes `IFLOW.md` with comprehensive project documentation. As of v0.2.6 (August 2025), `/init` no longer reads the `.iflow` folder contents by default, but does update an existing `IFLOW.md`.

### User-level configuration file

| Property | Value                    |
| -------- | ------------------------ |
| Filename | `settings.json`          |
| Format   | JSON                     |
| Location | `~/.iflow/settings.json` |

Example:

```json
{
  "theme": "Default",
  "selectedAuthType": "iflow",
  "apiKey": "your iflow key",
  "baseUrl": "https://apis.iflow.cn/v1",
  "modelName": "Qwen3-Coder",
  "searchApiKey": "your iflow key"
}
```

### Project rules file (PromptScript target)

| Property | Value                     |
| -------- | ------------------------- |
| Filename | `project.md`              |
| Format   | Plain Markdown            |
| Location | `.iflow/rules/project.md` |
| Schema   | None — free-form Markdown |

The `.iflow/rules/project.md` path used by the PromptScript formatter is consistent with the tool's use of a `.iflow` dot directory and a nested rules convention. However, it is **not explicitly documented** in official iFlow CLI documentation as a rules-loading path. The formatter adopts a plausible convention by analogy with other tools in the ecosystem.

## Supported Features

### Execution modes

iFlow CLI supports four permission modes that govern what the AI is allowed to do:

| Mode            | Description                                     |
| --------------- | ----------------------------------------------- |
| Default         | No permissions — read-only operations           |
| Plan mode       | Plan first, then execute with user confirmation |
| Accepting edits | File modification permissions only              |
| YOLO            | Maximum permissions — all operations allowed    |

### AI model support

- Qwen3-Coder, Kimi K2, DeepSeek v3 (free via iFlow Open Platform)
- Any OpenAI-compatible endpoint (configurable via `settings.json`)
- Claude and ChatGPT via OpenAI protocol compatibility

### Additional capabilities

- **SubAgent system**: Specialized sub-agents installable from the iFlow Open Market
- **Custom instructions marketplace**: Install custom instruction sets from the Open Market
- **MCP integration**: Model Context Protocol server support
- **Multimodal input**: Screenshot paste via Ctrl+V
- **Conversation history**: Automatic save and rollback; auto-compression at 70% context capacity
- **IDE plugins**: VS Code and JetBrains integrations
- **CI/CD**: GitHub Actions integration available
- **Blacklist check**: Rule-based detection of high-risk shell operations
- **AI review**: Context-aware risk analysis before execution

### Safety governance

The blacklist check uses cross-platform pattern matching to flag dangerous operations. The AI review layer applies context-aware analysis and selects an execution strategy. This means coding rules loaded into context can influence risk decisions.

## Multi-File Support

No hierarchical rules file discovery (analogous to Cursor's nested `.mdc` files or Amp's parent-directory AGENTS.md chain) was found in the official documentation. The CLI's primary project-context mechanism appears to be the single root-level `IFLOW.md` file produced by `/init`.

The PromptScript `multifile` and `full` versions would emit skill files under `.iflow/skills/<name>/SKILL.md`, which is consistent with the tool's dot-directory convention but is not verified against official documentation.

---

## Skills System

iFlow does not appear to have a native skills system with a defined file format equivalent to Amp's YAML front matter skills. Skills/custom instruction sets are distributed through the iFlow Open Market and installed via the CLI. No local skill file format specification was found in the documentation.

The PromptScript formatter uses `hasSkills = true` (the default from `createSimpleMarkdownFormatter`), which will emit `.iflow/skills/<name>/SKILL.md` files in `full` mode. Whether iFlow CLI auto-discovers these files is unconfirmed.

## PromptScript Formatter Assessment

### Current implementation

```typescript
// packages/formatters/src/formatters/iflow.ts
export const { Formatter: IflowFormatter, VERSIONS: IFLOW_VERSIONS } =
  createSimpleMarkdownFormatter({
    name: 'iflow',
    outputPath: '.iflow/rules/project.md',
    description: 'iFlow CLI rules (Markdown)',
    mainFileHeader: '# Project Rules',
    dotDir: '.iflow',
  });
```

### Assessment

| Aspect                      | Status          | Notes                                                                              |
| --------------------------- | --------------- | ---------------------------------------------------------------------------------- |
| Output filename             | Unverified      | `.iflow/rules/project.md` is plausible but not confirmed in official documentation |
| Format (Markdown)           | Correct         | iFlow reads Markdown; IFLOW.md is plain Markdown                                   |
| Dot directory               | Correct         | `.iflow` matches the tool's own convention for user/project config                 |
| Main file header            | Acceptable      | `# Project Rules` is functional; iFlow has no documented heading requirements      |
| `hasSkills` default         | Unverified      | Skill file auto-discovery by iFlow CLI is not documented                           |
| `hasAgents` / `hasCommands` | Default (false) | No native agents/commands sub-format documented for project files                  |
| Skill file naming           | Default         | Uses `SKILL.md` — no iFlow-native skill file format to conflict with               |

### Potential improvements

- **Verify rules file path**: If iFlow CLI adds formal documentation for project-level rule loading, the output path may need to align with an official location (e.g., root `IFLOW.md` rather than `.iflow/rules/project.md`).
- **Root IFLOW.md consideration**: The tool's primary project-context file is `IFLOW.md` at the project root. A future formatter version could optionally target that path for users who want their compiled rules to be picked up by `/init`-based workflows.
- **Skills discovery**: Once iFlow documents a local skill discovery mechanism, `hasSkills` configuration should be revisited.

---

## Conclusion

iFlow CLI is a Tier 3 formatter target. The PromptScript formatter uses a reasonable convention (`.iflow/rules/project.md`) that aligns with the tool's dot-directory pattern, but official documentation does not explicitly confirm this as a rules-loading path. The format (plain Markdown, no schema) is correct. The tool's primary project-context mechanism is `IFLOW.md` at the root, generated by `/init`, which is separate from the PromptScript-compiled output. No changes to the formatter source are required at this time; the output is valid Markdown that iFlow can accept as context, but automatic discovery of `.iflow/rules/project.md` should be confirmed against the tool's evolving documentation.
