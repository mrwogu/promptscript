## Research Validation

### Research Accuracy Assessment

The research report is **largely accurate** with one significant finding that requires correction and one gap that needs qualification.

**Confirmed correct:**

- Output path `INSTRUCTIONS.md` is correct. The report correctly identifies that OpenClaw does not have a single canonical instruction file equivalent to `CLAUDE.md`, and that `INSTRUCTIONS.md` as a project-level Markdown file fed verbatim into the context window is the appropriate choice.
- Registry name `openclaw` matches the export in `packages/formatters/src/formatters/index.ts` (`OpenClawFormatter`, `OPENCLAW_VERSIONS`).
- The formatter is registered and exported in the tier-3 section of the index, consistent with the research's Tier 3 classification.
- `dotDir: 'skills'` is confirmed in the source file. The research correctly flags this as potentially incorrect (see gaps below).
- Three output modes (`simple`, `multifile`, `full`) via `createSimpleMarkdownFormatter` are confirmed in the code.
- `mainFileHeader: '# INSTRUCTIONS.md'` is confirmed in the source.

**Gap requiring qualification — Skill YAML frontmatter:**

The research states the factory does not emit YAML frontmatter in skill files. This is **incorrect as of the current codebase**. `MarkdownInstructionFormatter.generateSkillFile()` (in `packages/formatters/src/markdown-instruction-formatter.ts`, lines 377–400) always emits:

```
---
name: <skill-name>
description: <skill-description>
---
```

This frontmatter is generated for all formatters that use `createSimpleMarkdownFormatter` with `hasSkills: true` (the default). The OpenClaw formatter inherits this behavior. The research gap about missing skill frontmatter is therefore **already resolved** in the current implementation.

**Gap requiring confirmation — `dotDir: 'skills'` path:**

The research correctly flags uncertainty about whether `skills/` (project-relative) is the right path versus `.openclaw/skills/`. Comparing against peer formatters:

- `windsurf.ts` uses `dotDir: '.windsurf'` → skills at `.windsurf/skills/<name>/SKILL.md`
- `opencode.ts` uses `dotDir: '.opencode'` → skills at `.opencode/skills/<name>/SKILL.md`
- `openclaw.ts` uses `dotDir: 'skills'` → skills at `skills/<name>/SKILL.md`

The `skills/` dotDir is anomalous. OpenClaw's documented workspace path is `~/.openclaw/workspace/skills/`, which is global, not project-relative. Whether a project-relative `skills/` directory is scanned is unconfirmed by the research sources. This remains the primary unresolved question.

**Workspace files scope confirmed correct:**

The research correctly concludes that `SOUL.md`, `AGENTS.md`, `TOOLS.md`, `USER.md`, `HEARTBEAT.md`, and memory files are workspace-layer concerns, not project-level instruction file concerns. The formatter intentionally does not generate these, and this is the correct scope boundary.

**Feature table accuracy:**

The feature table at rows 9–10 (`yaml-frontmatter`, `frontmatter-description`) are marked "No" for "Formatter Implements." These are incorrect given the finding above — skill files do receive YAML frontmatter. The feature table should show "Yes" for both rows when `hasSkills: true`. All other rows are accurate.

## Changes Required

### Priority 1 — Correct `dotDir` (Medium confidence required)

**Issue:** `dotDir: 'skills'` produces skill output at `skills/<name>/SKILL.md`. This path is unusual compared to all peer formatters, which use a hidden dot directory. There is no documented evidence that OpenClaw scans a project-relative `skills/` directory.

**Options:**

| Option      | Value                   | Output Path                                  | Notes                                               |
| ----------- | ----------------------- | -------------------------------------------- | --------------------------------------------------- |
| A (current) | `'skills'`              | `skills/<name>/SKILL.md`                     | Anomalous; no documented support                    |
| B           | `'.openclaw'`           | `.openclaw/skills/<name>/SKILL.md`           | Consistent with peer formatters; speculative        |
| C           | `'.openclaw/workspace'` | `.openclaw/workspace/skills/<name>/SKILL.md` | Mirrors global workspace; unusual for project-level |

**Recommendation:** Change to `dotDir: '.openclaw'` (Option B) to align with the pattern used by all other platform-specific formatters. This is a single-line change in `openclaw.ts`.

**File:** `packages/formatters/src/formatters/openclaw.ts`
**Change:** `dotDir: 'skills'` → `dotDir: '.openclaw'`

This also corrects the version description strings generated by `buildVersions()` in `create-simple-formatter.ts`, which currently produce misleading output like `INSTRUCTIONS.md + skills/skills/<name>/SKILL.md`.

### Priority 2 — No changes to skill frontmatter

The research identified missing YAML frontmatter as a high-priority gap. Code inspection confirms frontmatter is already emitted by `generateSkillFile()`. No code change is needed.

### Priority 3 — No changes to `INSTRUCTIONS.md` output

The primary output format (`# INSTRUCTIONS.md` header, plain Markdown body) is correct and matches OpenClaw's documented behavior. No changes needed.

### Priority 4 — Test coverage verification

Verify that the OpenClaw formatter has test coverage for:

- `simple` mode produces `INSTRUCTIONS.md` with `# INSTRUCTIONS.md` header
- `full` mode produces skill files at `skills/<name>/SKILL.md` with YAML frontmatter

If tests exist and cover these paths, no action required. If not, add a spec file at `packages/formatters/src/formatters/openclaw.spec.ts`.

## Implementation Notes

### What the Formatter Correctly Implements Today

1. `INSTRUCTIONS.md` as the primary output file with `# INSTRUCTIONS.md` H1 header.
2. Three output modes (`simple`, `multifile`, `full`) inheriting all standard section rendering from `MarkdownInstructionFormatter`.
3. Skill files with YAML frontmatter (`name:`, `description:`) under `skills/<name>/SKILL.md` in `full` mode.
4. Standard content sections: Project, Tech Stack, Architecture, Code Style, Git Commits, Config Files, Commands, Post-Work Verification, Documentation, Diagrams, Restrictions.
5. Correct export in `packages/formatters/src/formatters/index.ts` as a Tier 3 formatter.

### Known Platform Limitations (Not Formatter Bugs)

- OpenClaw does not support glob-based rule targeting — all content in `INSTRUCTIONS.md` applies globally.
- OpenClaw does not support slash commands, activation types, MDC format, workflow files, or nested directory rule structures. None of these require formatter changes.
- Workspace files (`SOUL.md`, `AGENTS.md`, `TOOLS.md`, etc.) are outside project-level scope and are intentionally not generated.
- There is no documented character limit for `INSTRUCTIONS.md`.
- `@`-mention include syntax is not documented for OpenClaw's `INSTRUCTIONS.md`; the formatter correctly does not emit any.

### Implementation Sequence

If proceeding with the `dotDir` correction:

1. Update `dotDir: 'skills'` to `dotDir: '.openclaw'` in `packages/formatters/src/formatters/openclaw.ts`.
2. Run `pnpm nx test formatters` and verify no regressions.
3. Run the full verification pipeline (`format`, `lint`, `typecheck`, `test`, `validate`, `schema:check`, `skill:check`).
4. Create a feature branch and PR per the project branching strategy.

### Research Gaps That Remain Open

The following items from the research could not be resolved against available documentation and remain open for future investigation:

1. **Project-scoped skill path**: Whether OpenClaw loads skills from a project-relative directory at all, and if so, what that path is. Resolution would require testing against a live OpenClaw installation or finding authoritative documentation.
2. **Mermaid diagram rendering**: Whether OpenClaw's Markdown rendering supports Mermaid fenced code blocks. This does not affect the formatter today (diagrams are rendered as fenced code blocks regardless).
