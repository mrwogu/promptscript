### Output path analysis

The research identifies that `.iflow/rules/project.md` is a plausible convention consistent with iFlow's `.iflow` dot-directory pattern but is not confirmed in official documentation. The iFlow CLI's primary project-context mechanism is `IFLOW.md` at the project root (generated by `/init`). This is a known ambiguity and does not constitute a bug in the current formatter — it is an accepted tradeoff for a Tier 3 target with incomplete public documentation.

### Skills system analysis

The research correctly notes that iFlow has no documented local skill discovery. The formatter emits `.iflow/skills/<name>/SKILL.md` files in `full` mode as a convention. This is structurally consistent with peer Tier 3 formatters (e.g., `adal`, `qoder`, `zencoder`) that share the same default `hasSkills = true` configuration with no native skills system confirmed. No action required.

### Registry registration

`IflowFormatter` and `IFLOW_VERSIONS` are exported from `packages/formatters/src/formatters/index.ts`. The formatter is registered and available to consumers. No gap found.

### Potential future changes (not required now)

These are deferred until official iFlow documentation is published or the tool's behavior is empirically verified:

1. **Output path — root `IFLOW.md`**: If iFlow documents that `IFLOW.md` at the project root is the canonical rules-loading path, the `outputPath` should change from `.iflow/rules/project.md` to `IFLOW.md`. This would be a one-line change to `outputPath` in `iflow.ts` and a corresponding update to `IFLOW_VERSIONS` descriptions.

2. **Skills discovery**: If iFlow documents a local skill discovery path (e.g., `.iflow/skills/`) and a skill file format, `hasSkills` configuration should be revisited. If iFlow uses a different skill file naming convention, `skillFileName` should be set explicitly.

3. **Sub-agent support**: iFlow's SubAgent system installs agents from the Open Market. If a local agent file format is documented, `hasAgents = true` and a custom `generateAgentFile` override would be needed — similar to the `FactoryFormatter` pattern used for Factory AI droids.

## Implementation Notes

### If output path changes to root `IFLOW.md`

The `outputPath` in `iflow.ts` would change to `'IFLOW.md'`. The `dotDir` would remain `'.iflow'` for skill/agent file emission. The `buildVersions` function in `create-simple-formatter.ts` already handles standalone file paths correctly via the `isNested` branch — no changes to the factory are needed.

```typescript
// Hypothetical change only — DO NOT apply until confirmed
export const { Formatter: IflowFormatter, VERSIONS: IFLOW_VERSIONS } =
  createSimpleMarkdownFormatter({
    name: 'iflow',
    outputPath: 'IFLOW.md', // changed from '.iflow/rules/project.md'
    description: 'iFlow CLI rules (Markdown)',
    mainFileHeader: '# IFLOW.md', // changed to match filename convention
    dotDir: '.iflow',
  });
```

### If skills discovery is confirmed

If iFlow confirms skill discovery under `.iflow/skills/` with `SKILL.md` files, no code change is needed — this is already the current behavior. If a different skill file name is used (e.g., `skill.md` or `instructions.md`), add `skillFileName: '<name>'` to the factory options.

### If sub-agent support is needed

Convert `iflow.ts` from a factory call to a class-based formatter extending `MarkdownInstructionFormatter`, following the pattern established in `packages/formatters/src/formatters/factory.ts`. Override `extractAgents` and `generateAgentFile` to match iFlow's SubAgent file format. Set `hasAgents: true` in the constructor config.
