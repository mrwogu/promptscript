### Research Finding Confirmed: Critical Path Bug

The research correctly identifies that `.junie/rules/project.md` is not an auto-discovered path by Junie. Junie discovers only:

1. `.junie/AGENTS.md` (preferred, current standard)
2. `.junie/guidelines.md` (legacy, still fully supported)
3. `AGENTS.md` (project root, CLI fallback)

The current output will be silently ignored by Junie unless the user manually sets a custom path in IDE settings — a significant usability defect for all existing users of the Junie formatter.

### Research Finding Confirmed: Header Naming Gap

The research correctly identifies that `# Project Rules` does not match Junie's own documentation and community catalog, which use `# Project Guidelines` or section-level H2 headers with no top-level H1. The change is low-risk and low-priority but correct.

### No Excess Features Found

No frontmatter, MDC format, glob patterns, slash commands, skills, or other unsupported features are currently generated by the Junie formatter. The simple formatter factory produces plain Markdown only, which is exactly what Junie expects.

## Implementation Plan

### Change 1 — Fix Output Path (Critical)

**File:** `packages/formatters/src/formatters/junie.ts`

Change `outputPath` from `.junie/rules/project.md` to `.junie/guidelines.md`.

Rationale for choosing `.junie/guidelines.md` over `.junie/AGENTS.md`:

- `.junie/guidelines.md` has been supported since Junie's initial release and is guaranteed to work with all JetBrains IDE plugin versions, including older installations.
- `.junie/AGENTS.md` is the Junie CLI's current preferred format, but IDE plugin support for this path may lag behind CLI behaviour in older plugin versions.
- The research recommends `.junie/guidelines.md` for broadest compatibility and notes that CLI users who prefer the `AGENTS.md` standard can rename or symlink the file after generation.
- All other team-committed Junie guidelines in the JetBrains community catalog use `guidelines.md`.

The change is a single-string edit to `outputPath`. Because `createSimpleMarkdownFormatter` derives the `buildVersions` descriptions directly from `outputPath`, all three version descriptions (`simple`, `multifile`, `full`) will update automatically without further changes.

**Before:**

```ts
outputPath: '.junie/rules/project.md',
```

**After:**

```ts
outputPath: '.junie/guidelines.md',
```

### Change 5 — Add Junie to Parity Matrix (Low)

**File:** `packages/formatters/src/parity-matrix.ts`

Add `junie` to the `requiredBy` or `optionalFor` arrays of each `SectionSpec` in `PARITY_MATRIX`, and add a `junie` entry to each `headerVariations` map. Because Junie is a plain-Markdown formatter with no format-specific section naming, all header variations match the default (`## Section Name`).

Junie is a plain content formatter — it has no required sections mandated by the platform. However, since it uses `addCommonSections()` from `MarkdownInstructionFormatter`, it will produce all the same sections as other simple formatters. Register it as `optionalFor` all standard sections, with no entries in `requiredBy`. This is consistent with how other Tier 2 formatters that use `createSimpleMarkdownFormatter` would be registered.

Header variations for all sections follow the `## {Section Name}` Markdown convention. Restrictions use `## Restrictions` (same as `opencode`, `gemini`).

## Out of Scope

The following items from the research are explicitly out of scope for this implementation:

- **MCP server config generation** — the `.junie/mcp/` directory is for configuring MCP servers, not for text guidelines. This is a different concern and falls outside the formatter's responsibility.
- **Multi-file folder format** (`.junie/guidelines/`) — the legacy folder format is still supported by Junie but the single-file format is equally valid. Implementing a folder-split mode would require a non-trivial formatter override and adds complexity without clear user benefit for the typical use case. Defer to a future enhancement if requested.
- **`AGENTS.md` output path** — `.junie/guidelines.md` is the correct choice for broadest compatibility. The `AGENTS.md` variant is not planned for this implementation.
