## Official Documentation

**Primary source:** https://docs.windsurf.com/windsurf/cascade/memories

Windsurf's AI assistant is called **Cascade**. Context persistence is handled through two mechanisms:

- **Memories** — automatically generated by Cascade from conversations; carry across sessions.
- **Rules** — manually defined markdown files; version-controlled, team-shareable, with explicit activation control.

Rules can be defined at three scopes:

| Scope               | Path                                                                                                                                            |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| Global              | `~/.codeium/windsurf/memories/global_rules.md`                                                                                                  |
| Workspace           | `.windsurf/rules/*.md` (current workspace and subdirectories, up to git root)                                                                   |
| System (Enterprise) | macOS: `/Library/Application Support/Windsurf/rules/*.md`; Linux/WSL: `/etc/windsurf/rules/*.md`; Windows: `C:\ProgramData\Windsurf\rules\*.md` |

An alternative legacy location also exists: `.windsurfrules` in the project root (single-file, no frontmatter).

Windsurf also supports **AGENTS.md** files (similar to root-level always-on rules, no frontmatter required) and **Workflows** (`.windsurf/workflows/*.md`, manually invoked via `/workflow-name`).

## Expected File Format

### Workspace Rules (`.windsurf/rules/*.md`)

Each rule is a standard Markdown file with optional YAML frontmatter declaring its activation mode.

```markdown
---
trigger: always_on
---

# Project Rules

- Rule content here
- Bullet points recommended
```

#### Frontmatter Fields

| Field         | Type           | Required                  | Description                                         |
| ------------- | -------------- | ------------------------- | --------------------------------------------------- |
| `trigger`     | string         | Yes (for workspace rules) | Activation mode (see below)                         |
| `globs`       | string or list | Conditional               | File patterns; required when `trigger: glob`        |
| `description` | string         | Recommended               | Summary shown to model for `model_decision` trigger |

#### Trigger / Activation Modes

| Trigger Value    | Behavior                                                                   |
| ---------------- | -------------------------------------------------------------------------- |
| `always_on`      | Full rule content included in system prompt on every message               |
| `model_decision` | Only `description` shown initially; full content loaded on demand by model |
| `glob`           | Applied when Cascade accesses files matching the `globs` pattern           |
| `manual`         | Excluded from system prompt; activated explicitly via `@rule-name` mention |

Global rules (`global_rules.md`) and root-level `AGENTS.md` files require **no frontmatter** — they activate automatically and are always on.

#### Glob Pattern Examples

```markdown
---
trigger: glob
globs: **/*.test.ts
---
```

Standard glob syntax is supported: `*.js`, `src/**/*.ts`, `**/*.test.ts`.

#### Character Limits

| Scope                          | Limit             |
| ------------------------------ | ----------------- |
| Global rules file              | 6,000 characters  |
| Workspace rule file (per file) | 12,000 characters |

Content exceeding per-file limits is truncated. Global rules take priority over workspace rules if the combined total would exceed context.

### Body Format

Documentation recommends:

- Bullet points and numbered lists over long paragraphs.
- XML tags for grouping related rules (e.g., `<coding_guidelines>...</coding_guidelines>`).
- Specific, concise rules — vague or generic rules are less effective.
- Standard Markdown headers (`#`, `##`) for section structure.

## Conventions and Best Practices

From official documentation:

1. **Use bullet points and numbered lists** — easier for Cascade to follow than long paragraphs.
2. **Be specific and concise** — overly long or vague rules reduce effectiveness.
3. **Avoid generic rules** ("write good code") — already baked into model training.
4. **Use XML tags** to group related rules (optional but documented as effective).
5. **Always declare a trigger** in workspace rule files — without it, activation behavior is undefined.
6. **Keep files under 12,000 characters** — content is truncated without warning beyond this limit.
7. **Prefer Rules for standards/conventions** and Memories for session-specific facts.
8. **Use AGENTS.md for simple location-based rules** and `.windsurf/rules/*.md` for rules requiring activation control.
9. **Version-control rule files** — they are designed to be committed to the repository and shared with the team.

## Gap Analysis

### Correct (implemented correctly)

- Output path `.windsurf/rules/project.md` matches the documented workspace rules directory.
- File format is standard Markdown, which is exactly what Windsurf expects.
- All standard content sections (project, tech stack, architecture, code standards, git, config, commands, post-work, documentation, diagrams, knowledge, restrictions) are rendered as plain Markdown — appropriate for this format.
- Skills output path `.windsurf/skills/<name>/SKILL.md` matches the expected Windsurf skills directory structure.
- Main file header `# Project Rules` is appropriate for a rules file.

### Incorrect (implemented differently from what platform expects)

- **No frontmatter is emitted.** Windsurf workspace rule files should declare a `trigger` field in YAML frontmatter. Without it, Cascade's behavior for the file is undefined — it may or may not be included depending on platform version. The file should at minimum include `trigger: always_on` to guarantee consistent activation equivalent to the current intent.

### Missing (platform supports but not implemented)

- **`trigger` frontmatter field** — must be `always_on`, `model_decision`, `glob`, or `manual`. Currently absent.
- **`description` frontmatter field** — used by Cascade when `trigger: model_decision` to decide whether to load the full rule. Useful for large rule files to avoid always consuming context.
- **`globs` frontmatter field** — enables file-pattern-based rule targeting. Not exposed from the PromptScript language to the Windsurf formatter. Cursor's formatter supports `@guards`-to-glob mapping; Windsurf's does not.
- **Workflow file generation** — Windsurf's `.windsurf/workflows/*.md` files provide named, manually-invoked step sequences. The formatter has `hasCommands: false` and no workflow output. The PromptScript `@shortcuts` block with multi-line content could map to workflow files, analogous to how the Cursor formatter maps them to `.cursor/commands/*.md` files.

### Excess (implemented but platform does not use / has no effect)

- The three-version model (`simple`, `multifile`, `full`) is inherited from the factory but `multifile` and `full` behave identically for this formatter (since `hasCommands: false` and skills only appear in `full`). The distinction is meaningful but the version descriptions in `buildVersions` may be misleading — `multifile` says "skills via full mode" rather than distinguishing any actual multifile behavior.

## Language Extension Requirements

To fully leverage Windsurf's rules format from PromptScript source files, the following language or formatter extensions would be needed:

1. **Rule trigger mode property** — A way to express activation mode in `.prs` source, e.g.:

   ```
   @targets {
     windsurf: {
       trigger: "always_on"  // or "model_decision", "glob", "manual"
     }
   }
   ```

   This would allow the formatter to emit the correct `trigger` frontmatter field.

2. **Rule description for model_decision** — A short description string expressible per-target or per-rule file for `model_decision` trigger mode.

3. **Glob pattern mapping** — The `@guards` block (already used by Cursor's multifile formatter) could be reused to generate glob-targeted `.windsurf/rules/*.md` files in a Windsurf multifile mode, parallel to Cursor's multifile implementation.

4. **Workflow mapping from shortcuts** — Multi-line `@shortcuts` entries could be emitted as `.windsurf/workflows/<name>.md` files (enabling `/name` invocation in Cascade), analogous to Cursor's `.cursor/commands/<name>.md` mapping. This would require enabling `hasCommands` or adding a `hasWorkflows` flag to the formatter config.

None of these require core language changes — they can be handled as formatter-level conventions using existing PromptScript blocks (`@targets`, `@guards`, `@shortcuts`).

## Recommended Changes

Listed in priority order:

### High Priority

1. **Emit `trigger: always_on` frontmatter** — The formatter should prepend YAML frontmatter to the output file:
   ```markdown
   ---
   trigger: always_on
   ---
   ```
   This is the minimum required to guarantee the rules file is always included in Cascade's system prompt, which is the current intent. Without it, activation is platform-version-dependent.

### Medium Priority

2. **Add workflow file generation** — Set `hasCommands: true` and implement a Windsurf-specific command/workflow generator that outputs multi-line `@shortcuts` entries as `.windsurf/workflows/<name>.md` files. Single-line shortcuts remain as inline documentation in the main rules file. This mirrors the Cursor formatter's command file generation and would unlock `/workflow-name` invocation in Cascade.

3. **Implement Windsurf multifile mode** — Add a genuine multifile variant that generates separate `.windsurf/rules/*.md` files for different glob patterns, using data from `@guards`. Each generated file would include appropriate `trigger: glob` frontmatter with the relevant `globs` pattern.

### Low Priority

4. **Add `description` frontmatter support** — For large projects where the 12,000 character limit is a concern, expose a way to emit `trigger: model_decision` + `description` frontmatter so Cascade loads the rule on demand rather than always consuming context window.

5. **Verify skill file frontmatter** — Windsurf's native skill system may expect specific frontmatter fields in `.windsurf/skills/<name>/SKILL.md` beyond the current `name` and `description` fields. Cross-reference against `docs.windsurf.com/windsurf/cascade/skills` to confirm compatibility.

6. **Add character limit warning** — The formatter has no awareness of Windsurf's 12,000-character-per-file limit. A compile-time warning when the output file exceeds this limit would prevent silent truncation in production.
